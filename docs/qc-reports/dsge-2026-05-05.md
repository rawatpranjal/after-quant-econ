## Findings

### Blockers

### Majors

**`dsge/nkdsge/model.mod` has Dynare syntax errors and undeclared
references.** The .mod file as committed:

- Line 1 declares only `var y pi i;` but line 5 (above the `model;` block)
  uses `r_nat`, `psi_y`, `rho_a`, `a`, `rho_z`, `z` — none declared as
  variables, varexos, or parameters.
- Line 18 (`i = rho+phi_pi*pi+phi_y*y+e`) is missing the trailing `;` that
  Dynare requires inside `model;`.
- Line 3 declares `k` as a parameter and line 10 sets `k = 0.95`. Then the
  `initval;` block on lines 22–28 treats `k`, `a`, `c` as variables — `k`
  conflicts with the parameter and `a, c` are not declared anywhere.
- Line 13 sets `se = 0.01` but `se` is not declared as a parameter in the
  parameters list (line 3).

If a reader tries to run `dynare model.mod`, it will fail to parse. The
tutorial works as written because run.py only reads the file as text for the
"Model Setup" section; it does not execute Dynare.

This is **not auto-fixed**. Two reasons:

1. `dsge/CLAUDE.md` says "`.mod` files contain model equations in Dynare
   syntax (kept as-is)" — a strong signal that the .mod is preserved as a
   legacy artifact, not maintained as a working spec.
2. Fixing the file requires choosing whether to (a) make it a working,
   determinate Dynare spec (which would change the numerical example) or
   (b) leave the indeterminate calibration but make the syntax parse. That
   choice is the user's.

If the goal is (b), a minimal patch would be: add `r_nat psi_y rho_a a rho_z z`
to the parameter/varexo lists with sensible default values; add the missing
`;` on line 18; replace the bogus `initval;` block with one over the declared
vars only; declare `se` as a parameter.

### Notes (not findings)

The reviewer initially flagged a second major — that the README does not
justify the calibration switch from `phi_pi=0.33, kappa=0.95` (in the .mod)
to `phi_pi=1.5, kappa=0.3` (in the run.py). On reread, **README.md line 70
already does this**:

> The source `model.mod` uses $\phi_\pi=0.33$ and $\kappa=0.95$. This report
> uses a standard determinate calibration, $\phi_\pi=1.5$ and $\kappa=0.3$,
> because the economic point is monetary transmission under a stable Taylor
> rule. The contrast matters: when policy fails to lean hard enough against
> inflation, the forward-looking system no longer selects a unique stable path.

That's a complete and correct explanation. Reviewer over-flagged.

## Suggested follow-up

1. Decide what to do with `dsge/nkdsge/model.mod`. Options:
   - Patch the syntax so the file is at least Dynare-parseable (keeps the
     legacy calibration but makes the file honest).
   - Replace it with a clean working spec that matches the calibration the
     run.py actually uses.
   - Add a one-line note at the top of the .mod (`// legacy spec; not
     intended to be executed; see README for the active calibration.`)
     and accept the syntax errors as noise.
2. The other two .mod files (`dsge/rbc/model.mod`, `dsge/assetNews/model.mod`)
   were not deeply audited — they were not flagged by static intake or by
   the reviewer, but a syntactic Dynare check (parsing in any Dynare
   version) would be a useful one-time verification step if you go down
   path 1 or 2 above.

## Postscript (2026-05-05, same day)

After this report was written, follow-up work landed:

- The folder was renamed `dynare/` → `dsge/`. No tutorial in this section
  invokes Dynare; the framing was misleading. Catalog section heading
  updated from "DSGE and Dynare" to "Linearized DSGE."
- `dsge/nkdsge/model.mod` syntax was repaired. All three `.mod` files now
  parse and gained a header note clarifying that `run.py` does not execute
  Dynare and pointing readers at the official install path if they want it.
- `lib/perturbation.py` was added: a Klein (2000) generalized-Schur (QZ)
  solver — the same algorithm Dynare uses for `stoch_simul, order=1` per
  Villemot (2011) "What Dynare Does." Each of the three tutorials now
  cross-checks its closed-form coefficients against this QZ solver; the
  three reported max-abs differences are 1.5e-15, 2.2e-16, and 2.2e-16.
- `dsge/rbc-with-labor/` was added as a fourth tutorial that uses Klein QZ
  as the primary solver — endogenous labor pushes the system past what
  hand-derived coefficients comfortably handle.
