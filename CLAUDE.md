# Computational Economics

## Project Overview
Pedagogical library of computational and structural economics models. The active
repo is a tutorial catalog, not a dump of course files. Every active model must
live in a self-contained folder with `run.py` that regenerates `README.md`,
`figures/`, and optional `tables/`.

The public name is `Computational Economics`: a tutorial collection for readers
who know the standard quantitative-economics foundations and want executable
structural and computational economics examples. It is fine to mention QuantEcon
as related background or inspiration, but do not imply official affiliation or
position this repo as piggybacking on that project.

## Catalog Ideology
- Organize the public catalog by economic subject first and numerical method
  second. Readers should be able to start from the economic question, then see
  the method used to answer it.
- Dynamic programming is a first-class subject in this repo, not merely a
  supporting method. Keep it near the top of the root README and order it from
  simple Bellman-equation examples to larger macro applications.
- Macroeconomics should be a coherent subject block covering heterogeneous
  agents, linearized DSGE, global nonlinear DSGE, continuous-time equilibrium,
  and optimal control. Do not scatter those examples only by solver family.
- Within each subject, list canonical and simple tutorials before advanced
  variants. A reader should see the progression, not a flat inventory.
- Use descriptive tutorial titles that name the economic object and topic. Avoid
  one-word or purely method titles when a more concrete title is possible.
- Keep method labels in catalog rows, but do not let method categories drive the
  main navigation.

## Tutorial Contract
Every tutorial folder must have:
- `run.py` as the single executable entrypoint.
- `README.md` generated by `lib.output.ModelReport`.
- `figures/thumb.png` for the root catalog.
- Reproducible outputs from running `python run.py` inside the folder.
- A root `README.md` catalog row with thumbnail, descriptive link title, what it
  teaches, method, and key insight.

The standard README section flow is:
Overview -> Equations -> Model Setup -> Solution Method -> Results -> Takeaway
-> References.

Do not pass a subtitle / description string to `ModelReport(...)`. Tutorials
have a title (`#`) and an Overview section; the blockquote line under the title
is removed and must not be reintroduced. New `run.py` files should call
`ModelReport("Title", include_reproduce=False, ...)` with no second positional
argument.

Do not use em-dashes (U+2014) or en-dashes (U+2013) anywhere in the repo,
including prose in `run.py` strings, generated `README.md` files, `CLAUDE.md`,
the style guide, scripts, and QC reports. Use a plain hyphen (`-`), a comma,
or rephrase. This applies to AI-generated reports as well; the proofread
prompt enforces this for new outputs.

Formal tutorials should use symbols where they make the model clearer. Include
symbols in `Model Setup` tables when the tutorial defines a formal state,
estimator, market, or equilibrium object. Write pseudocode in the same notation
when the method is mathematical, and keep reader-facing tables free of
code-style column names.

`Reproduce` is optional in the generated report. The folder contract still
requires `python run.py` to regenerate outputs, but the report does not need to
repeat that command when it adds clutter. Visible figure captions are also
optional when the surrounding prose already interprets the figure; keep useful
alt text and figure descriptions either way.

For prose tone, pseudocode, figures, tables, takeaways, and catalog-row wording,
see `STYLE_GUIDE.md`. It records soft stylistic conventions, not additional
validation rules.

## Code Style
- Python 3.11+.
- Prefer JAX (`jax.numpy as jnp`) for numerical kernels when it fits the model.
- NumPy/SciPy are acceptable for estimation, simulation, IO examples, and legacy
  translations where JAX would add clutter rather than clarity.
- Type hints on public helper functions.
- Google-style docstrings or concise equivalent docstrings.
- kebab-case folder names, snake_case file/function names.
- Keep each `run.py` focused on economics, computation, and report generation.

## Shared Library
- `lib/` contains shared utilities: grids, discretization, interpolation, VFI,
  STPFI, simulation, plotting, and output generation.
- Models import from `lib/` via:
  `sys.path.insert(0, str(Path(__file__).resolve().parents[2]))`
- Do not duplicate utility code in model folders when a shared helper already
  exists.
- If a second tutorial needs the same nontrivial helper, promote it to `lib/`.

## Output Standard
Every model `run.py` produces:
- `README.md`: auto-generated via `lib.output.ModelReport`.
- `figures/*.png`: matplotlib, 150 DPI, academic style.
- `tables/*.csv`: optional, if the model has tabular results.
- `figures/thumb.png`: 200x150 thumbnail for the root catalog.

After adding or changing tutorials, run:

```bash
python scripts/validate_catalog.py
```

## Math Rendering
- Edit math in `run.py`, regenerate the tutorial, and inspect the generated
  `README.md` before accepting the page.
- Prefer plain parentheses, square brackets, or `\lbrace ... \rbrace` for set
  braces. Avoid raw escaped curly braces and left/right curly-brace delimiters
  in generated Markdown.
- Avoid GitHub-fragile size-delimiter commands from the big-l/big-r family; use
  plain delimiters or `\left(...\right)` when needed.
- Do not emit display-math lines that begin with `+`, `-`, or another
  Markdown-list-like operator. Keep the operator on the previous line.
- Literal-star scripts should use `\ast` inside the script braces.
- Underbraces are allowed when they clarify economics, but keep each expression
  compact and inspect the rendered Markdown after regeneration.

## Legacy Policy
- Active topic folders should contain polished tutorial folders only.
- Original notebooks, PDFs, MATLAB/R scripts, and course handouts belong under
  `_legacy/`.
- Do not delete source material. Move it into `_legacy/` when cleaning active
  folders.
- New work should not depend on files in `_legacy/` at runtime.

## Do Not
- Do not create new Jupyter notebooks in active folders.
- Do not leave `.ipynb_checkpoints` outside `_legacy/`.
- Do not add interactive or Plotly visualizations. Use matplotlib only.
- Do not add dependencies beyond `requirements.txt` / `pyproject.toml` without
  discussion.
- Do not add a tutorial that cannot regenerate its README, figures, and tables
  from `python run.py`.
